Fetch the full record for a scan, including the triage report once the
scan has completed.

```http
GET /api/v2/scans/{scan_id}
```

## Path parameters

<ParamField path="scan_id" type="string" required>
  The `scan_id` returned by
  [`POST /api/v2/scan`](/api/scans/create) or one of the per-profile
  ids from a batch.
</ParamField>

## Response

The response is the scan record. Fields you should rely on:

<ResponseField name="profile_id" type="string" required>
  Same value as `scan_id`. Tumban tracks scans and profile records
  under the same identifier.
</ResponseField>

<ResponseField name="batch_id" type="string" required>
  The batch id. Scans submitted via [`POST /api/v2/scan`](/api/scans/create)
  get a synthetic `single_<scan_id>` batch id for internal bookkeeping;
  match on the `single_` prefix to distinguish single-scan submissions
  from batched ones.
</ResponseField>

<ResponseField name="url" type="string" required>
  The submitted profile URL.
</ResponseField>

<ResponseField name="status" type="string" required>
  See [Status values](/reference/status).
</ResponseField>

<ResponseField name="created_at" type="string" required>
  ISO 8601 UTC timestamp of when the scan was submitted.
</ResponseField>

<ResponseField name="processing_started_at" type="string">
  When pipeline processing actually began. `null` while waiting for a
  concurrency slot.
</ResponseField>

<ResponseField name="processing_completed_at" type="string">
  When pipeline processing finished. `null` while still processing.
</ResponseField>

<ResponseField name="callback_url" type="string">
  The callback URL the scan was submitted with (or the org default if
  not provided), as a string. Empty when no callback was configured.
</ResponseField>

<ResponseField name="metadata" type="object">
  The metadata you submitted with the scan, echoed back unchanged.
</ResponseField>

<ResponseField name="error" type="string">
  Present when `status` is `failed`. Brief description of what went
  wrong.
</ResponseField>

<ResponseField name="webhook_delivered_at" type="string">
  ISO 8601 UTC timestamp of successful webhook delivery. Absent when
  no callback was configured or delivery has not (yet) succeeded.
</ResponseField>

<ResponseField name="triage_report" type="object">
  Present once `status` reaches a terminal value with a usable result —
  today that means `completed` (`completed_with_partial` is
  [reserved](/reference/status) and not currently emitted). Shape below.
</ResponseField>

### `triage_report`

<ResponseField name="recommendation" type="string" required>
  One of `no_flags`, `review_low`, `review_medium`, `review_high`. See
  [Recommendation values](/reference/recommendation).
</ResponseField>

<ResponseField name="risk_score" type="integer" required>
  0–100. See [Recommendations](/concepts/recommendations).
</ResponseField>

<ResponseField name="confidence" type="string" required>
  `low`, `medium`, or `high`. See [Confidence](/concepts/confidence).
</ResponseField>

<ResponseField name="reason_codes" type="string[]" required>
  Machine-readable codes explaining the decision. See
  [Reason codes](/reference/reason-codes).
</ResponseField>

<ResponseField name="reason_summary" type="string" required>
  Human-readable summary of what was found.
</ResponseField>

<ResponseField name="review_targets" type="string[]" required>
  URLs your reviewers should look at first.
</ResponseField>

<ResponseField name="link_chain" type="string" required>
  Short label describing the path Tumban followed (e.g.
  `Profile -> External site`).
</ResponseField>

<ResponseField name="evidence_index" type="object[]">
  Per-URL evidence Tumban cited in support of the decision. See
  [Evidence index](/concepts/evidence-index).
</ResponseField>

## Coverage

The scan record's `triage_report` does **not** include `coverage` —
that field is delivered as a top-level key in the
[webhook payload](/webhooks/payload). When polling, read the
`coverage` object (delivered alongside the triage report on the webhook
side) to see what was skipped; today the scan's `status` is `completed`
even when individual steps failed (`completed_with_partial` is
[reserved](/reference/status) and not currently emitted).

## Errors

| Status | Detail |
|--------|--------|
| 404 | `Scan not found` — the scan id does not exist within your organization. |

## Using the dashboard

Click any non-processing scan row from the **Home** page or the
**Scan** page to open the scan detail. The page renders these
read-only sections:

- **Summary** — profile URL, scan id, status, submitted/completed
  timestamps, webhook delivery timestamp, error.
- **Triage** — recommendation badge, risk score, confidence, reason
  codes, reason summary.
- **Coverage** — which analysis steps ran, login-blocked URLs,
  referrer match counts.
- **Raw JSON** — collapsible *"Show full document"* panel that
  exposes the entire scan record.
